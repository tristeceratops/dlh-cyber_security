# The Vector-to-Asset Matrix

## Introduction

### Goal
Produce a systematic cross-reference showing which attack vectors can reach which critical assets, creating a complete threat exposure map.

### Context
Individual vectors are data points. Cross-referenced vectors are intelligence. A phishing email alone is a nuisance. A phishing email that delivers credentials for the flat network that reaches the EHR database that contains 50,000 patient records, that is a kill chain waiting to happen.

## Answer

### Vector-to-Asset Matrix

The matrix below cross-references the most plausible attack vectors against MedDefense's most critical assets. A cell is filled only when a realistic attack path exists based on the 1x00 asset registry, criticality assessment, gap analysis, and network scan summary.

| Attack Vector | [A-001 ehr-srv-01](../1x00_first_watch/7-asset_registry.md) | [A-002 ehr-db-01](../1x00_first_watch/7-asset_registry.md) | [A-005 ad-dc-01](../1x00_first_watch/7-asset_registry.md) / Active Directory | [A-021 MRI-CTRL-WS (WS-RAD-01)](../1x00_first_watch/7-asset_registry.md) | [A-013 FortiGate-100F](../1x00_first_watch/7-asset_registry.md) | Medical IoT | Active Directory |
|---|---|---|---|---|---|---|---|
| **Phishing / Spear Phishing** | Credential theft against a clinician or IT user leads to VPN or EHR sign-in, then lateral movement to ehr-srv-01 through the flat network and broad internal reachability. | Stolen EHR or VPN credentials are reused after phishing, allowing the attacker to reach ehr-srv-01, harvest application access, and then pivot to ehr-db-01 over the exposed database path. | Phishing a privileged user or helpdesk analyst can yield domain credentials, letting the attacker authenticate to Active Directory and expand into all connected services. | A radiology user phished with a fake vendor message can hand over credentials that are then used to access the MRI control workstation through normal remote support or desktop workflows. | Vendor-themed phishing aimed at IT can steal firewall admin or VPN credentials and allow direct changes to FortiGate policy or remote access settings. | A phish to clinical staff can expose device-management credentials or session tokens, which are then used to reach monitors, pumps, or nurse-call interfaces on the medical device network. | Executive or administrative phishing can steal Microsoft 365 credentials and then be used to reset or abuse Active Directory-linked accounts and trust relationships. |
| **VPN Exploit** | A compromised VPN appliance or stolen VPN session gives an attacker a foothold on the internal network, from which they can reach the EHR application server and begin reconnaissance or exploitation. | Once inside through VPN, the attacker can scan the flat network, reach PostgreSQL on ehr-db-01, and target the database directly because internal segmentation is absent. | VPN compromise often leads to credential harvesting and domain discovery, letting the attacker target domain controllers and Active Directory authentication paths next. | A VPN foothold can expose internal Radiology systems, including the legacy MRI control workstation, because the flat network does not isolate the imaging lane. | The firewall/VPN layer itself is the attack target here; an exploit or weakly protected management surface can allow policy changes, tunnel creation, or boundary bypass. | A VPN foothold can be used to scan and access pump, monitor, and nurse-call management interfaces because these systems sit on the same internal reachability plane. | A VPN exploit gives the attacker broad internal access, which is often enough to reach and abuse Active Directory-backed authentication across the hospital. |
| **Default / Shared Credentials** | Shared clinician or support credentials can be used to log into ehr-srv-01 without accountability, especially if those credentials are harvested from a low-friction internal target. | Shared access or reused passwords can let an attacker move from ehr-srv-01 into ehr-db-01, where the lack of strong internal containment makes database abuse easier. | Default or reused domain credentials are a direct route to domain-level access, especially when passwords are weak or accounts are retained after offboarding. | A shared radiology account can authenticate to the MRI control workstation and hide who performed the action, which is especially dangerous on a legacy host. | If firewall or remote-access administrative credentials are shared or weak, an attacker can alter perimeter rules, create tunnels, or disable controls. | Default device logins on pumps and monitors can be used to alter device settings, harvest telemetry, or establish a foothold inside the medical-device environment. | Shared passwords on domain-connected services can be reused to reach Active Directory or to impersonate legitimate users across dependent systems. |
| **Vulnerable Software Exploit** | Exploiting outdated software on public-facing or internally reachable hosts can yield code execution on ehr-srv-01, which is a strong pivot point into the clinical stack. | A vulnerability on an adjacent host can be used to steal database credentials or directly attack ehr-db-01 because the database is broadly reachable. | Once one vulnerable system is compromised, the attacker can harvest credentials or abuse trust paths to reach Active Directory and expand control. | The Windows XP MRI workstation is the clearest vulnerable-software path: a legacy exploit can compromise the host and then use its network reach to pivot deeper into Radiology or other internal systems. | Vulnerabilities in the firewall, VPN stack, or management interface can collapse the boundary entirely and expose the rest of the network. | Vulnerable embedded web services or outdated device firmware can be exploited to control or destabilize infusion pumps, monitors, or nurse-call systems. | Exploiting a vulnerable workstation or server often leads to credential theft, which can then be used to access Active Directory through normal authentication. |
| **Supply Chain Compromise** | A vendor compromise affecting EHR support or remote maintenance can be used to reach ehr-srv-01 through trusted administrative channels. | If the EHR application server is compromised via a vendor path, the attacker can pivot to ehr-db-01 and steal or alter the underlying patient record store. | Compromise of an IT or identity vendor can be leveraged to authenticate into or poison Active Directory trust relationships and recovery workflows. | A compromised medical imaging or device vendor can target the MRI control workstation because that system depends on trusted support relationships and legacy tooling. | A vendor or managed-service compromise involving network equipment could directly affect the firewall policy plane and open the perimeter from the inside. | Medical device vendors often have privileged maintenance channels, so a supply-chain compromise can reach pumps, monitors, or nurse-call controllers with little resistance. | A compromised cloud or identity vendor can be used to abuse directory-linked accounts and hybrid authentication across MedDefense. |
| **Insider (Malicious)** | A malicious insider with legitimate EHR access can exfiltrate records or sabotage the application tier, then use that access to move toward more privileged systems. | A database or application administrator can intentionally query, copy, or alter ehr-db-01 data and hide activity inside routine support work. | A disgruntled administrator or helpdesk worker can abuse their internal access to weaken or disable Active Directory, creating widespread authentication failure or unauthorized access. | Radiology insiders with legitimate access can misuse the MRI control workstation or its workflow path to steal imaging data or interfere with studies. | A privileged insider can alter FortiGate rules, VPN settings, or routing to create covert access paths or blind the boundary. | A clinical engineer or biomed insider can intentionally modify device settings or open management interfaces on pumps and monitors. | A malicious insider may abuse directory administration rights to add accounts, reset passwords, or preserve access after termination. |
| **Insider (Negligent)** | An inattentive clinician leaving a session open or using a shared workstation can expose ehr-srv-01 through unauthorized viewing or accidental actions. | A careless user can connect to ehr-db-01 indirectly through exposed application paths or misrouted internal tooling and accidentally trigger data exposure. | Manual offboarding and weak access review mean a terminated or confused user may still have valid Active Directory access for days or weeks. | A radiology user who shares credentials or fails to log out can leave the MRI control workstation exposed to misuse without an obvious audit trail. | A tired IT administrator can misconfigure the firewall or VPN while responding to a rushed request, inadvertently widening the attack surface. | Unmanaged endpoints and poor training make accidental interaction with device management pages or unsafe USB use more likely on the medical device network. | Negligent users often keep using old passwords or shared logins, which can sustain valid directory access even when the behavior is sloppy rather than malicious. |
| **Physical Access** | Physical access to a clinical workstation or server room can be used to capture credentials or directly reach ehr-srv-01 from a trusted internal position. | A person with physical access can plug into the internal network, map the flat environment, and attempt database access to ehr-db-01 from inside the perimeter. | Physical access to server rooms or networking closets can expose domain controllers, cached credentials, and password-reset opportunities that lead to Active Directory compromise. | Physical access to Radiology is especially dangerous because the MRI workstation is legacy and patient-critical; an attacker can tamper with the host, insert media, or observe the control process. | Direct access to the network closet or firewall area can expose the FortiGate management plane and allow policy changes, port mirroring, or device resets. | Physical access is one of the easiest ways to abuse medical IoT, because device consoles, USB ports, and local admin interfaces are often available on the ward. | Physical access to a workstation or badge-controlled area can be converted into directory compromise through cached sessions, password capture, or local logon abuse. |

### Most Connected Assets

1. **[A-005 ad-dc-01 / Active Directory](../1x00_first_watch/7-asset_registry.md)** — This is the most connected asset because nearly every vector can be turned into a directory compromise path, and directory control amplifies access across the rest of the environment.
2. **[A-013 FortiGate-100F](../1x00_first_watch/7-asset_registry.md)** — The perimeter device is reachable through phishing, VPN abuse, vulnerable software, supply chain compromise, insider misuse, and physical access, making it the main boundary intersection.
3. **[A-001 ehr-srv-01](../1x00_first_watch/7-asset_registry.md)** — The EHR application server is reachable through most identity, credential, and internal-lateral-movement paths, so it sits at the center of patient-care compromise.

### Most Versatile Vectors

1. **Phishing / Spear Phishing** — This is the most versatile vector because it can deliver credentials, malware, or social proof that opens access to the EHR, Active Directory, the firewall plane, and even medical-device workflows.
2. **Insider (Negligent)** — This vector reaches many assets because ordinary mistakes, shared accounts, and unattended sessions can expose clinical, identity, and administrative systems without any technical bypass.
3. **VPN Exploit** — This vector is highly versatile because one external compromise can land directly inside the flat network, from where nearly every critical asset becomes reachable.
